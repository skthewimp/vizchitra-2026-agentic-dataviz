## Possible workshop arc

  ### Part 1: What is a good visualization?

  Purpose: establish taste + standards.

  Not just good/bad examples. Ask:

  - What is the chart trying to say?
  - Does it say it clearly?
  - What work is the reader forced to do?
  - What choices are doing useful work?
  - What choices are decoration?

  This grounds “beautiful” in function.

  ### Part 2: Your chart, your story

  Before LLMs do anything, participants articulate:

  - dataset
  - audience
  - question
  - possible story
  - why this matters

  This prevents “AI, explore my data and find magic.”

  ### Part 2.5: Let LLM make first-cut chart

  Important pedagogically. Let them see the failure.

  The first chart becomes evidence:
  This is what happens when you skip the system.

  Then you can critique:

  - wrong/weak story?
  - bad chart type?
  - no annotation?
  - generic title?
  - wrong colour hierarchy?
  - unsupported claim?

  ### Part 3: Visualization as scientific process

  This is the conceptual heart.

  Frame:

  > A chart is a claim with evidence arranged visually.

  So process is:

  hypothesis → data → test → revise → communicate

  LLMs help at every stage, but cannot own the judgment.

  ### Part 4: Narrative chart

  Good section after scientific process.

  A good chart should be:

  - self-sufficient
  - opinionated
  - specific
  - annotated
  - honest about limits
  - designed around one main reading path

  Key line:

  > If the reader has to hunt for the insight, the chart is not doing its job.

  ### Part 5: Build the system

  This becomes the practical workflow:

  1. import / clean data
  2. understand schema
  3. generate hypotheses
  4. validate hypotheses
  5. select main narrative
  6. choose visual form
  7. specify annotations / labels / hierarchy / colours
  8. make chart
  9. critique rendered output
  10. refine once or twice

  ### Part 6: LLM help at each step

  This is where you go “agentic”.

  Not “LLM makes chart”, but:

  - LLM as data profiler
  - LLM as hypothesis generator
  - LLM as skeptical analyst
  - LLM as visual editor
  - LLM as copywriter for titles/annotations
  - LLM as style enforcer
  - LLM as reviewer

  ### Part 7: Production workflows

  This is the bridge from "I made one better chart" to "this can actually run again."

  Not production in the sense of:

  > AI magically publishes a dashboard while everyone goes for coffee.

  More like:

  > The workflow has briefs, standards, checks, owners, and refresh rules.

  Useful pieces:

  - style guide as system input
  - chart request brief
  - visual QA gate
  - tool-specific handoff
  - refresh notes
  - escalation rules

  The same system should be able to produce:

  - R / Python chart code
  - Tableau or Power BI implementation notes
  - Datawrapper / Flourish instructions
  - dashboard QA checklist
  - next-month refresh instructions

  Key question:

  > If 50 people in your organisation used this next month, what would you want standardized, banned, reviewed, or escalated?

  This also makes the "agentic" bit less hand-wavy.

  An agent is not just a clever prompt. It is the combination of:

  - context
  - taste
  - task brief
  - evidence checks
  - chart generation
  - review
  - memory for the next run

  Production means deciding which of these are automated, which are reviewed, and which still need human judgement.

  So:

  1. Good vs bad visualization
  2. Participants define their chart problem
  3. LLM generates naive first cut
  4. Critique first cuts
  5. Visualization as scientific process
  6. Narrative chart principles
  7. Build the system
  8. Use LLMs at each system step
  9. Regenerate/refine chart
  10. Turn the workflow into a production system
  11. Package reusable workflow

  That gives emotional payoff:

  They experience failure → then you give framework → then they improve.
